## Why HEBBS?

Every memory solution in the market shares the same architecture: Python glue, an external vector database, and LLM calls in the operational path. They converge to the same features because they draw from the same well -- and they share the same hard ceiling on latency, correctness, and operational simplicity.

HEBBS plays a different game. Cognitive operations are **algorithms inside a single embedded engine**, not LLM-call wrappers over a distributed stack:

- **Sub-10ms p99 recall** at millions of memories -- no network hop to an external vector DB
- **Crash-safe writes** via RocksDB WAL and atomic WriteBatch -- kill the process at any point and recover to a consistent state
- **Decay, importance, and reinforcement** computed as math, not LLM prompts
- **Four recall strategies** (similarity, temporal, causal, analogical) as bounded index traversals
- **One binary, zero external dependencies** -- deploy, monitor, and reason about a single process

## Core Operations

| Group | Operations | Purpose |
|-------|-----------|---------|
| **Write** | `remember`, `revise`, `forget` | Store, update, and prune memories |
| **Read** | `recall`, `prime`, `subscribe` | Retrieve context via multiple strategies |
| **Consolidate** | `reflect_policy`, `reflect`, `insights` | Background learning and knowledge synthesis |
